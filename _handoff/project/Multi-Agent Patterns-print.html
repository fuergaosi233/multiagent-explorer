PATTERN 01 · CENTRALIZED CONTROL

Supervisor / Manager Agents-as-tools · 主控代理把专家当作工具调用

别名 · ALIASES主控 agent / manager / subagents-as-tools / centralized orchestration
CORE MECHANISM · 核心机制
一个主 agent 保留最终控制权，把其他专业 agent 当作工具调用；子 agent 的中间过程通常不进入用户对话。

适合 · FIT
生产系统 · 客服分诊 · 内部 CLI 工具调用 · 代码任务中"主 agent + reviewer/tester/searcher" 的结构。
不适合 · UNFIT
需要真正自治协商、多个 agent 同权决策，或者不希望中心节点成为瓶颈的场景。
优点 · PROS
控制强、可观测、上下文隔离、容易加权限和审计。
风险 · RISKS
主 agent 路由错误会导致全局错误；复杂任务中主控可能成为 token 和延迟瓶颈。

PATTERN 02 · CENTRALIZED CONTROL

Router / Handoff / Transfer 路由器 · 专家接管 · transfer_to_xxx

别名 · ALIASES路由器 / 转交 / transfer_to_xxx / specialist 接管
CORE MECHANISM · 核心机制
当前 agent 根据任务类型把控制权转交给另一个 agent；后续交互由被转交的 agent 接管。

适合 · FIT
多领域客服 · 企业流程分流 · 审批 / 报销 / 技术支持 · 需要长期由某个 specialist 维护上下文的场景。
不适合 · UNFIT
只需一次性调用专家工具；或上下文不应暴露给被转交 agent 的场景。
优点 · PROS

风险 · RISKS
容易循环交接；handoff 判定需要约束；跨 agent 的上下文裁剪和权限传递比较关键。

Hierarchical Manager-Worker 公司式层级 · 树状组织 · 多级 manager

别名 · ALIASESDirector–manager–worker 树 / 多级嵌套 / corporate hierarchy
CORE MECHANISM · 核心机制
Agent 按层级组织，上级负责规划 / 分派 / 审核，下级负责执行；可多层嵌套。

PATTERN 03 · FLOW

Sequential Pipeline / Chain 顺序链 · 流水线 · DAG 的线性特例

别名 · ALIASESchain / pipeline / linear DAG / 流水线
CORE MECHANISM · 核心机制
多个 agent 按固定顺序执行，前一步输出作为后一步输入；每一步有明确输入输出契约。
Input
Planner
Researcher

draft
适合 · FIT
结构化流程 · 文档生成 · 代码生成流水线 · 固定审批链 · 稳定的企业 SOP。
不适合 · UNFIT
任务高度动态、需要中途重规划或多个 agent 自主协商时。
优点 · PROS
确定性高；调试简单；容易重放和做 checkpoint。
风险 · RISKS
上游错误会级联；整体延迟为各步骤之和；对变化适应性弱。

PATTERN 04 · FLOW

Parallel Fan-out / Fan-in Scatter-gather · concurrent orchestration · map-reduce

别名 · ALIASESscatter-gather / concurrent orchestration / map-reduce / 并行分发-聚合
CORE MECHANISM · 核心机制
同一个任务或拆分后的子任务并行发给多个 agent，再由 aggregator 汇总、投票或合成。
Input
Dispatcher
Security

Aggregator
适合 · FIT
多视角评审 · 并行检索 · 多模型对比 · 代码审查 · 安全 / 性能 / 可读性分别检查。
不适合 · UNFIT
子任务强依赖顺序，或共享状态容易冲突且无锁 / 隔离机制时。
优点 · PROS

风险 · RISKS
聚合器质量关键；并发状态写入可能冲突；成本可能显著上升。

PATTERN 10 · FLOW

Blackboard / Shared Workspace 黑板 · 共享记忆 · 事件总线 · 发布 / 订阅

别名 · ALIASESblackboard / shared memory / event bus / pub-sub
CORE MECHANISM · 核心机制
Agent 不一定直接对话，而是读写共享空间；当前共享状态决定下一步哪个 agent 行动。
Blackboard
shared state
Retriever
Data Agent
Reasoner
Verifier
适合 · FIT
异步研究 · 长期任务 · 数据湖检索 · 多 agent 贡献局部证据 · 需要共享状态和复盘的系统。
不适合 · UNFIT
缺少权限 / 版本控制；共享状态不可控；数据污染难以回滚。
优点 · PROS
天然异步；支持长期积累；每个 agent 可以按能力自主贡献。
风险 · RISKS
需要强 schema、版本、锁、TTL 和 provenance；否则会变成脏上下文池。

23落地路线 · ADOPTION ROADMAP
四阶段落地 From a controllable main loop to a cross-ecosystem agent network
PHASE 01

可控主链路

Controllable main loop
  • Supervisor + tools / subagents
  • Sequential task plan
  • 工具注册 · 权限 · 日志 · trace
  • 预算控制 · 可重放执行
PHASE 02

并行质量层

Parallel quality layer
  • Fan-out reviewer
  • Critic / judge 聚合
  • 安全 · 性能 · 正确性 · 规范
  • 并行 reviewer → 主 agent
PHASE 03

长任务与状态

Long-running tasks & state
  • Task store + blackboard
  • Worktree / sandbox isolation
  • 多轮开发 · 后台任务
  • 代码隔离 · session 恢复
PHASE 04

跨系统互联

Cross-system interop
  • MCP for tools / resources
  • A2A / ANP for agent-agent
  • 标准化能力发现
  • 外部 agent 调用 · 审计

24结论 · TAKEAWAYS

13 个模式 5 个维度 · 4 阶段路线

生产系统优先 Supervisor / Handoff / Sequential / Fan-out · Blackboard；质量增强用 Debate / Role-playing；跨生态互联用 A2A / MCP / ANP。
PRODUCTION

先做对，再做多

从 Supervisor + Sequential 起步；把可观测、权限、预算、可重放放在能力扩张之前。

QUALITY

用并行换质量

Fan-out reviewer 与 Debate 是廉价的质量杠杆；判定器质量决定整个 ensemble 的上限。

INTEROP

协议层迟早要面对

当 agent 与工具跨越组织 / 供应商边界，MCP · A2A · ANP 决定治理复杂度，越早设计越好。
