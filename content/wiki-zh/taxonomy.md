# 分类体系：五维视角

不要按框架名称对多智能体模式进行分类。按工程维度进行分类。任何真实系统通常同时组合多种模式 —— 例如：`监管者 + 并行扇出 + 黑板 + 验证器 + 人工审批 + MCP`。

## 1. 控制结构

| 类型 | 控制权所在 | 代表模式 | 适用场景 |
|---|---|---|---|
| 集中式 | 一个协调器/管理者拥有流程 | 监管者、智能体即工具、路由器 | 稳定的生产流、支持分流、企业任务 |
| 转移 | 活动智能体可以移交控制权 | 交接 | 多领域专家切换、长对话 |
| 层级 | 管理者管理工作器；工作器可以进一步分解 | 层级 | 大型项目、跨团队、多阶段工作 |
| 去中心化 | 没有固定中心；智能体自行交谈或中继 | 点对点、群体 | 开放探索、自主网络 |

## 2. 信息流

| 类型 | 信息如何移动 | 代表模式 |
|---|---|---|
| 线性 | A 的输出馈入 B，B 的输出馈入 C | 顺序管道 |
| 并行 | 多个智能体并发运行；结果合并 | 并行扇出 / 收集 |
| 共享线程 | 多个智能体在一个线程中发言 | 群聊 |
| 内部会话 | 智能体触发内部子对话 | 嵌套对话 |
| 共享状态 | 智能体不直接聊天；它们读写工作空间 | 黑板 |
| 事件流 | 通过主题/队列/事件总线通信 | 事件总线 / 发布-订阅 |

## 3. 决策制定

| 类型 | 如何做出决策 | 代表模式 |
|---|---|---|
| 管理者选择 | 协调器选择下一步 | 监管者、路由器 |
| 审查者判断 | 批评者/验证器评分质量 | 生成器-批评者 |
| 迭代 | 循环直到满足条件 | 精炼循环 |
| 对抗 | 正方/反方辩论；裁判得出结论 | 辩论 / 裁判 |
| 群体 | 多个智能体投票或集成 | 投票 / MoA |
| 市场 | 智能体竞价、定价或协商 | 市场 / 合同网 |

## 4. 执行环境

| 类型 | 重点 | 代表模式 |
|---|---|---|
| 角色驱动 | 智能体拥有专业角色和 SOP | 角色扮演 / 虚拟公司 |
| 环境隔离 | 智能体在独立工作空间/沙盒中运行 | 工作空间隔离 |
| 人机耦合 | 人类是特殊智能体 —— 批准、纠正 | 人在回路 |
| 环境媒介 | 智能体通过修改共享环境来协作 | 环境媒介协作 |
| 模拟学习 | 智能体在模拟环境中交互或训练 | 社会模拟 / MARL |

## 5. 协议互联

| 协议 | 谁连接到谁 | 用途 |
|---|---|---|
| MCP | 智能体 ↔ 工具 / 资源 / 提示 | 工具、数据、提示的标准化暴露 |
| A2A | 智能体 ↔ 智能体 | 跨框架、跨供应商的智能体协作 |
| ACP / ANP | 智能体 ↔ 智能体 / 网络 | 智能体通信、身份、网络发现 |
| 智能体客户端协议 | IDE / 客户端 ↔ 编码智能体 | 编辑器与编码智能体之间的标准化链接 |

## 结语

不要问"哪种多智能体模式最强" —— 而是问：

- 任务需要分解吗？
- 子任务可以并行运行吗？
- 专家需要拥有对话吗？
- 需要审查和回滚吗？
- 需要跨系统互操作吗？
- 需要长期状态和可观测性吗？

如果答案大多是"否"，暂时不要考虑多智能体。拥有正确工具和正确上下文的单个智能体通常更可靠。
