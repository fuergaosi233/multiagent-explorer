title: 治理、权限与成本
description: 工作流是工程系统而非聊天功能。审批、权限、成本控制、恢复、可观测性、回滚必须在长任务前设计好。
---

派生数十个写权限智能体、运行数小时、消耗大量 token 的工作流不是聊天交互，是生产工程操作，需照此对待。

## 运行前审批

启动前用户应看到：阶段列表、成本估计、Claude 生成的脚本原文。读脚本要像读生产库迁移脚本一样仔细：阶段边界是否符合预期？扇出形状是否匹配任务范围？是否有停止条件？是否写文件，写哪些路径？

## 权限继承

子智能体继承父会话的工具 allowlist。`bash`、`edit`、MCP、Web 等能力都会传递。运行前应：
1. 检查当前 allowlist；
2. 评估工作流范围是否值得这么宽的 allowlist；
3. 必要时缩小 allowlist。

即便 allowlist 已批准，部分命令仍可能触发中途权限弹窗，会阻塞工作流。运行前预先批准预期工具类别。

## 成本控制

| 控制 | 用法 |
|---|---|
| 缩小范围 | 先跑 5 个文件再跑 500 |
| 低风险阶段用小模型 | 发现/映射不必用与综合同等模型 |
| Agent 上限 | 限制并发 agent 数 |
| Token 预算 | 在停止条件中表达预算阈值 |
| 停止条件 | 收敛检查而非单纯预算耗尽 |

经验法则：未在小范围跑过的工作流不要直接跑全库。

## 恢复

同一会话内可基于 checkpoint 恢复；跨会话不保证。每个有显著成本的阶段后 checkpoint；pipeline 中每个 item checkpoint；存足够状态以便重建剩余阶段的扇出形状。设计时让阶段重入是安全的。

## 可观测性

应在工作流、阶段、agent、claim、checkpoint 多粒度记录。`/workflows` 提供进度视图，必要时把 trace 事件路由到自有监控系统。

## 回滚

写文件的工作流必须先设计回滚方式：worktree 隔离、PR 输出、人审 gate、dry-run 模式。任何写文件的工作流至少要在 worktree 或分支上跑，不要直接打到 main。

## 管理控制

平台管理员可在 managed Claude Code 环境中禁用动态工作流或按用户/项目限制使用。具体能力以最新 Claude Code 文档为准。
