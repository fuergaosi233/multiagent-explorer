> 数据来自 Anthropic 关于动态工作流和 Claude Opus 4.8 的官方公告，作为说明大规模工程任务结构的示例，数字与阶段以官方表述为准。

## 摘要

- 生成约 **75 万行 Rust 代码**；
- **99.8% 现有测试通过**；
- **11 天**从首个 commit 到可合并状态。

这不是单 agent 重写完成的，而是多阶段工作流：每阶段都有清晰输入、可验证输出和独立审查。

## 阶段拆解

### 阶段 1：Lifetime Mapping
先用工作流跨代码库映射 Zig 的所有权与生命周期语义，识别能直接翻译和需要人工干预的构造，产出每文件复杂度映射，决定阶段 2 的扇出形状。跳过该阶段会生成本地编译却集成失败的 Rust。

### 阶段 2：按文件做行为等价移植
worker agent 并行翻译，目标是行为等价而不是 idiomatic。每个 agent 输入一个 Zig 文件，输出能通过原有模块测试的 Rust。脚本强制并行边界，worker 不共享上下文。

### 阶段 3：每文件 reviewer agent
reviewer 不是"批准者"而是"找问题者"，检查 Zig 与 Rust 实现的语义差异，标记行为等价但脆弱的构造。

### 阶段 4：Build / Test 修复循环
build 失败路由到 fix agent，test 失败路由到能检查测试和代码的 agent，循环直到通过率达标或预算耗尽。大规模翻译第一遍不会全对，迭代必须是工作流的一等组成。

### 阶段 5：合并后清理 PR 工作流
主迁移合入后，单独的工作流删除已完整移植的 Zig 副本、更新 build 配置，并提 PR 让人审批后再合入。

## 工程教训

- **大规模迁移是阶段，不是一个提示**：每阶段有明确输入、可验证输出、独立审查；
- **验证是结构，不是可选项**：没有 reviewer 与 build/test 循环，99.8% 通过率不可能达到；
- **产出物是 PR 不是对话**：对话是触发器，工程交付是分支上的代码；
- **人审 gate 不可省**：自动化做到了能自动化的部分，但最终合并仍需人审。

## 风险

| 风险 | 说明 |
|---|---|
| 语言语义差异 | Zig 与 Rust 内存/错误模型不同，测试级等价不保证全条件下语义等价 |
| 测试覆盖空洞 | 99.8% 通过率指现有测试，翻译引入的新失败模式可能未覆盖 |
| reviewer 相关性 | 若 reviewer 提示词相似，可能共享系统性盲区 |
| 生成代码可维护性 | 行为等价的翻译未必 idiomatic，长期维护依赖后续重构 |
| 人审范围 | 75 万行无法逐行审，质量主要由工作流各验证阶段把关 |
